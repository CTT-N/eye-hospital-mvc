Integration Approach

How should the frontend HTML connect to the backend Java servlets?

A

HTML → JSP Conversion (Server-Side Rendering)

Convert each fe/*.html file into a .jsp file. Controllers forward to the JSP, which uses JSP EL (${data}) to render real data from the database.

Pros

  • Fits existing architecture perfectly
  • No new technology needed
  • Simplest to implement
  • Works with existing AuthFilter/RoleFilter

Cons

  • Full page reload on every action
  • JSP mixes HTML + Java logic
B

REST API + fetch() (Client-Side Rendering)

Add JSON API endpoints (e.g. /api/patient/appointments) to the backend. Update JS files to call fetch() and populate the DOM dynamically.

Pros

  • No page reloads (smoother UX)
  • Frontend stays as clean HTML
  • API reusable for future mobile app

Cons

  • Need to build entire API layer
  • Auth requires session or JWT handling in JS
  • Significant rework of all JS files
  • More complexity
C

Hybrid (JSP shell + fetch() for data)

JSP pages handle auth and render the page shell. Dynamic table/list data is fetched via lightweight JSON endpoints after page load.

Pros

  • Auth stays server-side (secure)
  • Dynamic sections feel responsive
  • Balanced complexity

Cons

  • Two patterns to maintain
  • More code than pure SSR

💡 My recommendation: Option A. Your backend is already Servlet/JSP — converting HTML to JSP is the natural path with the least new complexity. The existing AuthFilter and RoleFilter work automatically.
